# Git Policy — Хөгжүүлэлтийн багийн дүрэм

---

## Ерөнхий зарчим

### Тохиргоо (нэг удаа хийнэ)

Репо clone хийхээс өмнө доорх тохиргоог хийсэн байх ёстой:

```bash
git config --global user.name "Өөрийн нэр"
git config --global user.email "your_email@example.com"
```

---

## Branch дүрэм

- Онцлог, засвар, томоохон өөрчлөлт бүрт **тусдаа branch** үүсгэнэ.
- Branch нэр **тодорхой, утга бүхий** байх ёстой.

### Branch нэрлэх формат

| Төрөл | Формат | Жишээ |
|-------|--------|-------|
| Онцлог | `feature/тайлбар` | `feature/login-page` |
| Засвар | `fix/тайлбар` | `fix/issue-123` |
| Сургалт (үндсэн) | `trainee/nt-[нэр]` | `trainee/nt-bat` |
| Сургалт (дасгал) | `trainee/nt-[нэр]/тайлбар` | `trainee/nt-bat/fetch-api` |

> **Чухал:** Сургалтын trainee бүр `nt-` угтвартай username ашиглана — жишээ нь GitHub username нь `bat` бол branch нэр `trainee/nt-bat` байна.

- Branch нэрэнд **зөвхөн жижиг үсэг** болон `-` ашиглана.
- Хэтэрхий ерөнхий нэр (`fix`, `update`, `changes`) хэрэглэхгүй.

---

## Commit дүрэм

- **Байнга commit хий** — том нэг commit биш, жижиг утга бүхий commit-үүд.
- Commit мессеж нь **юу хийснийг биш, яагаад хийснийг** тайлбарлана.

### Commit мессежийн формат

```
<төрөл>: <товч тайлбар>
```

| Төрөл | Хэзээ ашиглах |
|-------|---------------|
| `feat` | Шинэ онцлог нэмэх |
| `fix` | Алдаа засах |
| `refactor` | Логик өөрчлөхгүйгээр кодыг сайжруулах |
| `docs` | Баримтжуулалт өөрчлөх |
| `test` | Тест нэмэх/засах |
| `chore` | Тохиргоо, хэрэгсэл өөрчлөх |

**Жишээ:**
```
feat: хэрэглэгчийн нэвтрэх хуудас нэмэх
fix: fetch алдааны боловсруулалтыг засах
docs: README-д суулгах зааврыг нэмэх
```

**Зайлсхийх зүйлс:**
- `update`, `fix stuff`, `changes` гэх мэт утгагүй мессеж
- Нэг commit-д хамааралгүй олон өөрчлөлт нэгтгэх

## Pull Request дүрэм

- PR нь **нэг зорилготой** байна — нэг онцлог, нэг засвар.
- PR нээхдээ **өөрийгөө assignee** болгоно.
- Feedback ирэхэд **шуурхай** хариулна.

### PR тайлбарт заавал оруулах зүйлс

PR template автоматаар ачаалагдана. Доорх бүх хэсгийг гүйцэд бөглөнө:

| Хэсэг | Агуулга |
|-------|---------|
| **Ерөнхий мэдээлэл** | Юу хэрэгжүүлсэн, ямар дасгал |
| **Юу өөрчлөгдсөн** | Файл, хавтас, үндсэн зан төлөв |
| **Checkpoint нотолгоо** | Screenshot, Live URL, PR линк |
| **Саад → Шийдэл** | Тулгарсан саад, шийдсэн арга, сургамж |
| **Шалгах жагсаалт** | Merge хийхийн өмнө бүгдийг тэмдэглэнэ |

---

## Code Review дүрэм

- PR бүрийг **дор хаяж нэг хүн** хянана.
- Хянах үед **кодын чанар, уншигдах байдал, стандарт** дагасан эсэхэд анхаарна.
- Томоохон зөрчил шийдсэний дараа **шинэ review хүснэ**.
- Reviewer approve хийхдээ inline comment үлдээж болно — trainee тэр comment бүгдийг шийдсэний дараа дахин review хүснэ.

---

## Merge & Rebase дүрэм

### Merge хийх

- **Зөвхөн approve болсны дараа** merge товчийг дарна.
- Merge нь өөрчлөлтийн тодорхой бүртгэл үлдээдэг тул **merge-ийг rebase-ийн оронд ашиглана**.
- Branch-аа **тогтмол update** хийж үндсэн branch-тай синк барина.

```bash
git checkout trainee/nt-bat
git pull origin trainee/nt-bat
git checkout trainee/nt-bat/fetch-api
git merge trainee/nt-bat
```

### Rebase хэзээ зөвшөөрөгдөх вэ?

PR нь **henüz review болоогүй** үед үндсэн branch-тай sync хийхэд rebase хэрэглэж болно. Гэхдээ болгоомжтой байх хэрэгтэй — commit history өөрчлөгдөнө.

```bash
git rebase trainee/nt-bat
```

> ⚠️ **Review болсон PR-д rebase хэзээ ч хийхгүй.**

---

## Branch хамгаалалт

- `main` branch-д **шууд push хориотой** — зөвхөн PR-ээр merge хийнэ.
- `trainee/nt-[нэр]` branch-д **Buddy зөвшөөрөлгүйгээр merge хийхгүй**.

---

## CI / Автомат шалгалт

PR нээх бүрт GitHub Actions автоматаар:

1. **Хавтасны бүтэц** шалгана (`validate-structure.sh`)
2. CI алдаатай бол merge хийх боломжгүй болно

CI алдаа гарвал лог уншиж алдааг засаад дахин push хийнэ.

---

## Сайн дурын зөвлөмж

### Кодын стандарт
- Багийн зөвшилцсөн стандарт, style guide дагана.
- Нэршлийн convention, comment, баримтжуулалтын best practice дагана.

### Тест
- Кодын өөрчлөлтийг хамарсан **unit test** бичнэ.
- PR нээхийн өмнө тестийг ажиллуулж, бүгд pass болсон эсэхийг шалгана.
